Calendar app — PyInstaller onefile bundle guide

This guide describes how to build and deploy the Windows onefile executable for the Google Calendar and contacts automation project. It contains no API keys, tokens, or secret filenames beyond generic placeholders.

Antivirus (e.g. Norton) and the “properties file not found” dialog

The message you saw lists two locations that were checked for calendar_api_properties.properties: the folder next to your .exe, then the PyInstaller temp folder under %LOCALAPPDATA%\Temp\_MEI…. That behavior matches the application’s config search order, not a generic “blocked by antivirus” failure.

Norton can sometimes quarantine a new or unsigned .exe, block first launch, or scan the _MEI extract folder aggressively. In those cases you might see a Norton alert, a failed start, or missing files if Norton removed something from a property_files folder.

Norton is unlikely to be the sole cause when the dialog explicitly says the properties file is missing and tells you to create property_files beside the executable. The fix is to add that folder and your real config files (created outside this guide), then run again. Optionally add an antivirus exclusion for your install folder or the built .exe after you trust it, and review Norton’s Security History if behavior is still suspicious.

What you are building

The entry script is google_calendar_and_contacts_automate.py. PyInstaller reads google_calendar_and_contacts_automate.spec and produces a single-file GUI executable named calendar_automate.exe under the dist\ folder.

Onefile mode unpacks dependencies to a temporary directory at runtime (_MEI…). Optional data files are only present there if they were available on the machine that ran PyInstaller when the bundle was built.

Runtime configuration (property_files)

At runtime the app looks for a directory named property_files in this order (frozen build):

  1. Next to the .exe (recommended for secrets you do not want embedded in the binary).
  2. Inside the PyInstaller unpack tree (only if those files were bundled at build time).

Typical files (names only — supply your own content offline)

  • calendar_api_properties.properties — application settings (non-secret keys may still live here; treat the file as sensitive).
  • google_desktop_oauth_client_contacts_api.json — OAuth client configuration from Google Cloud Console.
  • calendar-automate-srvc-account-ref-file.json or equivalent — service account material if you use that mode.
  • token.pickle — may be created after first OAuth sign-in; frozen builds often persist the token under user %APPDATA% (see project oauth_setup.py).
Do not commit secrets. Keep property_files out of version control. This HTML file intentionally avoids real paths to secrets and real key material.

Optional: add TASKS_AUTOMATION_ROOT=<path> to calendar_api_properties.properties (same line format as other keys). If the process environment variable TASKS_AUTOMATION_ROOT is already set, it takes precedence and the property is not applied.

Laundry, YouTube, and TASKS_AUTOMATION_ROOT

The onefile calendar_automate.exe is not a Python interpreter. The GUI must not launch satellite tools with sys.executable pointing at a .py file. Laundry and YouTube actions therefore resolve a real tasks_automation checkout on disk and start Windows .bat files in a new console (same pattern as double-clicking the batch file).

calendar_app_paths.resolved_tasks_automation_root() picks the first root that contains Laundry_TryCents\run_laundry.bat, trying in order:

  1. Environment variable TASKS_AUTOMATION_ROOT (if set).
  2. <folder of your .exe>\tasks_automation (optional sidecar layout).
  3. On Windows, a documented example path under C:\E_Drive\project_workspace\tasks_automation — replace with your own layout using the env var or property on other machines.

What gets launched (frozen, Windows)

  • Laundry: …\Laundry_TryCents\run_laundry.bat with working directory Laundry_TryCents (the batch uses %~dp0 and may call WSL or other tools defined inside your repo — keep the checkout intact).
  • YouTube Transcribe / Cleanup transcripts: …\YouTube_Transcribe\run_transcribe.bat or prune_transcripts.bat with working directory YouTube_Transcribe.
Tip: Keep the full tasks_automation tree (including Laundry_TryCents and YouTube_Transcribe) on the same PC as the installed .exe, or set TASKS_AUTOMATION_ROOT in the shortcut / system environment to that root.
Prerequisites
  • Windows with Python 3.12+ (examples use the py -3.14 launcher; adjust the tag to match your installed version).
  • For frozen installs, a full tasks_automation checkout on disk (sibling folders Laundry_TryCents and YouTube_Transcribe under that root) so Laundry / YouTube buttons can run their .bat files. The PyInstaller spec may still bundle a single laundry_automation.py for legacy paths; the GUI prefers run_laundry.bat from the resolved repo root.

Deploy

After a successful build, copy dist\calendar_automate.exe from the google_calendar_and_contacts_automate folder to your chosen install directory (for example a “Daily startup” applications folder). Place a property_files directory in the same folder as the executable unless you intentionally embed those files at build time.

Example copy (adjust paths): Copy-Item -Force dist\calendar_automate.exe "C:\…\calendar_api\calendar_automate.exe"

Tip: If Windows reports the file is in use, rename the new binary (for example calendar_automate_new.exe) or close running instances before overwriting. The latest build can be copied as a new name (for example calendar_automate_rebuilt.exe) and renamed after you close the old process.
Troubleshooting
Symptom Typical cause What to do
ModuleNotFoundError at startup Build interpreter missing packages Install the same imports your script uses, then rebuild with that interpreter.
Properties file not found (paths listed) No property_files beside the .exe and nothing bundled Create property_files next to the .exe and add your properties and JSON files.
Executable will not start; Norton alert Heuristic / reputation block Review Security History; restore if quarantined; consider exclusion for trusted build output.
Laundry / YouTube button does nothing or “not found” TASKS_AUTOMATION_ROOT not set and default path does not contain Laundry_TryCents\run_laundry.bat Set the env var or add TASKS_AUTOMATION_ROOT to properties; ensure the repo tree exists at that root.
Laundry used to fail silently with onefile sys.executable was the .exe, not python.exe Current builds launch run_laundry.bat via cmd.exe; rebuild after updating the Python sources.
